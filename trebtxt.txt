Введение
Целью проекта является разработка Telegram-бота, который помогает пользователям следить за рынком криптовалют и принимать инвестиционные решения.
Основные функции
1. Настройка предпочтений: выбор криптовалюты по умолчанию.
2. Настройка предпочтений: выбор фиатной валюты по умолчанию.
3. Отображение текущей цены выбранной криптовалюты.
4. Отображение истории цены отслеживаемых криптовалют за указанный период.
5. Сравнение цен двух криптовалют.
6. Добавление актива в портфель (с указанием количества).
7. Удаление актива из портфеля (с указанием количества)
8. Вычисление стоимости портфеля в данный момент времени.
9. Отслеживание общей стоимости портфеля во времени.
10. Расчёт и отображение изменения стоимости по каждой криптовалюте в портфеле.
11. Получение от LLM инвестиционного анализа по конкретной криптовалюте.
12. Получение от LLM обзора портфеля с рекомендациями.
13. Запросы к LLM-советнику в свободной форме о крипторынке.
14. Установка ценовых алертов на определенные криптовалюты.
15.  Автоматические уведомления при изменении цены более, чем на 5% для отслеживаемых криптовалют.
16.  Автоматические уведомления при превышении ценой установленного значения для криптовалюты из отслеживаемых валют.

Общие требования
Контроль версий и сотрудничество
Весь код должен быть размещен в репозитории GitHub (https://github.com/Java-Team-Spring-Framework/Cryptocurrency-Investment-Advisor-Bot).
Репозиторий должен содержать README.md файл с подробным руководством по:
* Локальной настройке проекта.
* Сборке и запуску приложения.
* Развертыванию приложения с помощью Docker.
* Взаимодействию с ботом Telegram.
История изменений должна отражать значимый вклад всех членов команды.


Целевые артефакты
* Приложение должно создавать работающий файл fat JAR в качестве основного артефакта.
* Приложение должно быть доступно для развертывания в виде контейнера Docker.
* Для управления зависимостями, сборки и упаковки приложения используется Gradle.

Функциональные требования
Аутентификация пользователей
Идентификация происходит по Chat ID, дополнительная регистрация не требуется.
Настройки взаимодействия
* Выбор пользователем фиатной валюты для представления цены криптовалюты:
○  Если пользователь не указал фиатную валюту, добавлять фиатную валюту по умолчанию (доллары США).
○  Представление цены криптовалюты в виде фиатной валюты устанавливается единым образом для всех активов, используемых для отслеживания.
○  Выбор фиатной валюты предоставляется из заранее определенного списка, который включает евро, японскую иену, фунт стерлингов, турецкую лиру, российский рубль, китайский юань. 
○  Если данная валюта уже выбрана, выдавать соответствующее сообщение и прекратить процесс выбора фиатной валюты.
* Просмотр текущей фиатной валюты для представления цены криптовалюты.
* Выбор пользователем криптовалют для отслеживания
○ Добавление криптовалюты в отслеживаемые
* Если пользователь не добавил ни одну криптовалюту для отслеживания, то в отслеживаемых указывается криптовалюта по умолчанию (BTC).
* Выбор криптовалют предоставляется из заранее определенного списка, который включает BTC, ETH, SOL, XRP, ADA, DOGE, AVAX, NEAR, LTC.
*  Если добавляемая валюта уже находится в отслеживании, выдавать соответствующее сообщение.
* При добавлении валюты пользователь указывает значение цены, за единицу криптовалюты при превышении которого ему отправляется уведомление. 
○ Удаление криптовалюты из отслеживаемых
               -   Выбор криптовалюты для удаления предоставляется из списка отслеживаемых на данный момент криптовалют. 
               -   Если после удаления в отслеживаемых не остается ни одной криптовалюты, то выдается соответствующее сообщение и в отслеживаемые добавляется криптовалюта по умолчанию (BTC).
* Просмотр текущего списка отслеживаемых криптовалют.
* Просмотр ответа на вопрос в свободной форме LLM-консультанту о криптовалютных рынках.
* Просмотр инвестиционного анализа на основе LLM для конкретной криптовалюты.
Взаимодействие с рыночными данными
* По запросу пользователя показывать текущую цену выбранной криптовалюты в ранее выбранной фиатной валюте. 
* По запросу пользователя показывать отчет об изменении цены отслеживаемых криптовалют в фиатной валюте за указанный период :1 день, 7 дней, 30 дней. 
* По запросу пользователя сравнивать цены на две выбранные криптовалюты.
Управление портфелем
* Добавление актива в портфель с указанием: тикера криптовалюты и количества монет
* Удаление актива из портфеля с указанием: тикера и количества монет. Выбор актива для удаления предоставляется из списка текущих активов в портфеле.
* Показывать содержимое портфеля с количеством монет и стоимости каждого актива.
* Вычислять общую стоимость портфеля на данный момент
* Отслеживать изменение стоимости портфеля за выбранный период: год, месяц, сутки
* Показывать изменение стоимости каждой криптовалюты в портфеле с момента первого добавления валюты в портфель.
* Получение обзора портфеля с рекомендациями LLM.
История портфеля
* Хранить следующую информацию о покупке актива:
* Дата в формате YYYY-MM-DD HH:MM:SS.
* Название криптовалюты.
* Цена в валюте по умолчанию. 


Уведомления
Уведомления – сообщения, несущие информацию о изменениях в отслеживаемом активе, которые могут побудить пользователя к действиям на рынке (покупке или продаже).
●          Посылать уведомления пользователю, когда:
○   Цена за единицу криптовалюты превышает установленное пользователем при значение для данной криптовалюты. 
○          Цена криптовалюты изменяется более, чем на 5% вверх или вниз за последние 24 часа. Уведомления начинают присылаться автоматически при добавлении криптовалюты в отслеживаемые.
●          Формировать и выводить список актуальных уведомлений для пользователя на данный момент.
●          Удалять уведомления из списка отправленных при потере актуальности уведомлений, то есть по истечении срока в 7 дней.
Источник данных для курса криптовалют
Сайт Binance (https://www.binance.com/en).

Нефункциональные требования
Производительность
* Запросы, которые зависят от внешних сервисов, должны выполняться неблокирующе: пользователь получает подтверждение приема запроса, а результат отправляется по мере готовности.
* Оптимизация базы данных: Структура базы данных PostgreSQL должна проектироваться с использованием необходимых индексов (например, по chat_id), чтобы обеспечивать быстрый поиск пользователей и их настроек без длительных блокировок.
Надежность
* Отказоустойчивость при работе с внешними API: При временной недоступности сторонних сервисов (API криптовалютной биржи Binance, LLM API, Telegram API) система должна использовать механизм повторных запросов (Retry) с возрастающей задержкой (например: 1с, 2с, 4с).
* Гарантия доставки: В случае сетевого сбоя при отправке уведомления пользователю, задача не должна теряться. Сообщение должно возвращаться в очередь брокера сообщений для повторной попытки отправки.
Безопасность
* Все конфиденциальные данные (токен Telegram-бота, ключи LLM API, учетные данные базы данных и брокера сообщений) должны передаваться исключительно через защищенные переменные окружения. Хранение ключей в исходном коде запрещено.
* Все данные, поступающие от пользователя (названия тикеров, числовые значения цен, объемы активов), должны проходить строгую валидацию по регулярным выражениям и диапазонам допустимых значений (например, отсутствие отрицательных сумм покупки).
Роли и доступы
В системе предусмотрены две разделенные роли:
1. Пользователь (User): Клиент системы, взаимодействующий с ней через интерфейс Telegram-бота. Имеет доступ только к своим рыночным предпочтениям, личному портфелю, истории запросов к LLM и своим уведомлениям. Идентификация происходит по chat_id.
2. Администратор (Admin): Пользователь с привилегированными правами, взаимодействующий с системой напрямую через вызовы HTTP API. Имеет доступ к системным метрикам и обезличенным данным всех пользователей.
Требования к авторизации
* Авторизация через Telegram (Роль: User):
   * Дополнительная регистрация с вводом логина/пароля не требуется.
   * При первой отправке команды /start система извлекает уникальный chat_id пользователя и создает новую запись в базе данных.
   * Все последующие команды бота проверяют наличие данного chat_id в системе. Если пользователя нет в БД, запрос отклоняется с предложением отправить /start.
* Авторизация через HTTP (Роль: Admin):
   * Осуществляется путем передачи токена доступа в HTTP-заголовке Authorization (формат: Bearer <API_TOKEN>).
   * Токен администратора должен быть надежно сгенерирован и задан на этапе развертывания через переменные окружения.
Конечные точки (HTTP Endpoints)
Сервер должен предоставлять следующие REST API эндпоинты для административного управления и мониторинга:
1. Проверка работоспособности системы
* Метод и путь: GET /healthcheck
* Авторизация: Не требуется (публичный доступ).
* Описание: Эндпоинт для мониторинга состояния приложения.
* Формат ответа: JSON-объект, содержащий текущий статус сервера (например, {"status": "UP"}) и массив с именами авторов-разработчиков проекта.
2. Получение списка пользователей
* Метод и путь: GET /users
* Авторизация: Строго требуется (только для роли Admin). Запросы без валидного заголовка Authorization должны получать статус 401 Unauthorized.
* Описание: Возвращает информацию о зарегистрированных пользователях системы.
* Формат ответа: Массив JSON-объектов, содержащий базовые данные пользователей (внутренний ID, дата регистрации, установленные валюты по умолчанию и т.д.).

Требования к технологическому стеку
* Telegram API: Приложение должно быть интегрировано с Telegram API для обеспечения функциональности бота.
* LLM API: Приложение должно быть интегрировано с LLM API  для обеспечения функциональности на базе искусственного интеллекта (OpenRoute).
* Язык программирования: Java 25  
* Фреймворк: Spring 7 (ReactiveSpring (WebFlux), JOOQ, Spring Rest Docs)
* База данных: PostgreSQL 18.3
* Контейнеризации приложения: Docker, Docker Compose
* Логирование: SLF4J
* Тестирование: JUnit

Сценарий использования


UC-1. Аутентификация
	Триггер
	Команда    /start
	Основной сценарий
	1. Пользователь отправляет команду  /start.
2. Бот устанавливает фиатную валюту по умолчанию, в отслеживаемые добавляется криптовалюта по умолчанию.
3. Выводится информация об установлении данных по умолчанию.
	



UC-2. Настройка фиатной валюты
	Триггер
	Команда   /set_fiat
	Основной сценарий
	1. Пользователь отправляет команду /set_fiat
2. Бот выводит сообщение о предложении выбора валюты. 
3. Пользователю предлагаются для выбора кнопки с доступными фиатными валютами или кнопки назад.
4. Пользователь выбирает новую валюту из предложенных.
5. Бот меняет старую валюту на новую и выводит информацию о смене фиатной валюты.
	Альтернативный сценарий
	     1-3 Основной сценарий
      4. Пользователь выбирают валюту, которая уже установлена.
      5. Бот выводит информацию о том, что данная валюта уже установлена.


	Альтернативный сценарий
	1-3 Основной сценарий
4. Пользователь выбирает кнопку “назад”
5. Бот прекращает процесс настройки фиатной валюты.
	



UC-3. Просмотр настроек фиатной валюты
	Триггер
	Команда /current_fiat
	Основной сценарий
	1. Пользователь отправляет команду /current_fiat
2. Бот выводит информацию о текущей фиатной валюте.
	



UC-4. Добавление криптовалюты в отслеживаемые
	Триггер
	Команда   /add_tracked_crypto
	Основной сценарий
	1. Пользователь отправляет команду /add_tracked_crypto
2. Бот выводит сообщение о предложении выбора криптовалюты для добавления. 
3. Пользователю предлагаются для выбора кнопки с доступными криптовалютами или кнопки “назад”.
4. Пользователь выбирает криптовалюту.
5. Бот подтверждает, что криптовалюты еще нет в отслеживаемых.
6. Бот отправляет сообщение с просьбой ввести цену в фиатной валюте, при превышении которой будет отправляться уведомление.
7. Пользователь вводит численное значение цены в фиатной валюте.
8. Бот проверяет корректность ввода и сохраняет данные отслеживаемой криптовалюты.
9. Выводится сообщение о добавлении криптовалюты и перечень криптовалют в отслеживаемых.


	Альтернативный сценарий
	     1-3 Основной сценарий
      4. Пользователь выбирает криптовалюту, которая уже есть в отслеживаемых.
      5. Бот выводит информацию о том, что данная валюта уже есть в отслеживаемых. Также выводится значение пороговой цены данной валюты.
	Альтернативный сценарий
	    1-6 Основной сценарий
    7. Пользователь вводит не численное значение.
    8. Бот выводит информацию о том, что введена некорректная валюта, и не добавляет новую запись в отслеживаемые.
    9. Бот прекращает процесс добавления новой криптовалюты.
	Альтернативный сценарий
	1-3 Основной сценарий

Глоссарий
Основные термины


Telegram 
	кроссплатформенный мессенджер, в котором пользователи обмениваются сообщениями и взаимодействуют с ботами.
	Telegram-бот
	программное приложение, работающее внутри Telegram и автоматически обрабатывающее команды и сообщения пользователей.
	Chat ID
	уникальный числовой идентификатор пользователя или чата в Telegram, используемый системой для идентификации и аутентификации пользователя.
	Пользователь
	лицо, взаимодействующее с ботом через Telegram и использующее его функции для получения информации о криптовалютах, управления портфелем и настройки уведомлений.
	Администратор 
	пользователь с расширенными правами доступа, который взаимодействует с системой через HTTP API и имеет доступ к административным функциям.
	Криптовалюта 
	цифровой актив, использующий криптографические методы для обеспечения безопасности транзакций и учета владения.
	Фиатная валюта
	государственная валюта, используемая для отображения стоимости криптовалют и результатов расчетов в приложении.
	Курс криптовалюты
	текущая стоимость криптовалюты, выраженная в выбранной фиатной валюте.
	Портфель 
	совокупность криптовалютных активов пользователя с указанием количества и цены покупки, используемая для расчета текущей стоимости, прибыли и убытка.
	Алерт
	пользовательское уведомление о наступлении определенного рыночного события, например достижения заданной цены или сильного изменения курса криптовалюты. 
	LLM-консультант
	модуль системы, использующий LLM API для генерации инвестиционного анализа.

	реляционная база данных, используемая для хранения данных.
	RabbitMQ 
	брокер сообщений, обеспечивающий асинхронное взаимодействие компонентов системы, например при обработке уведомлений и фоновых задач. 
	Аутентификация 
	процесс установления личности пользователя.
	Авторизация 
	процесс проверки прав доступа пользователя к определенным функциям системы после успешной аутентификации
	

Процесс разработки


Логирование 
	процесс записи событий и действий приложения для мониторинга, диагностики ошибок и анализа работы системы.
	Валидация
	проверка входных данных на корректность и соответствие заданным ограничениям.
	Развертывание 
	процесс подготовки и запуска приложения в целевом окружении.
	Gradle 
	система сборки проекта
	Docker 
	платформа для контейнеризации приложения и его зависимостей, обеспечивающая переносимость и единообразие окружения.
	Docker Compose
	инструмент для запуска многоконтейнерного приложения, позволяющий поднимать связанные сервисы, например приложение, базу данных и брокер сообщений, в локальной среде.
	Docker-образ 
	шаблон приложения, на основе которого создается Docker-контейнер для развертывания системы.
	JUnit 
	 библиотека для модульного тестирования Java-приложений, предусмотренная в требованиях к проекту
